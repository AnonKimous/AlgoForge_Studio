# AlgorithmScheduler 流水线模型

## 1. 目标

这份文档描述流水线算法在当前项目里的组织方式，以及调度中心应该暴露什么能力。

核心前提是：

- `Agent` 明确知道自己提交的是流水线算法
- 流水线算法对外只暴露两种同步语义：**强制同步** 和 **非强制同步**
- `Agent` 只关心提交了多少轮资源、最终结果是什么，不关心内部 lane 的具体身份
- 流水线的调度、推进、统计、挂载和卸载，都由 `AlgorithmScheduler` 统一管理

## 2. Agent 视角

从 `Agent` 的角度看，流水线算法有这些特征：

- 它是一个流水线算法
- 它支持强制同步和非强制同步
- 它会在调度中心里形成多条 lane
- `Agent` 只知道自己提交了几轮资源
- `Agent` 不追踪具体是哪一条 lane 在出口返回

`Agent` 看到的是算法对象，不是流水线内部实现。

## 3. 两种同步语义

### 3.1 强制同步

强制同步是面向调试和可视化的语义。

它的重点是：

- 让调度过程更容易观察
- 暴露单 stage 耗时
- 暴露整条流水线的总耗时
- 这些统计只给 `debugTool` 使用

强制同步不改变算法对外的提交形式，但会保留运行时统计信息，方便排查卡点和性能问题。

### 3.2 非强制同步

非强制同步是面向吞吐的语义。

它的重点是：

- 不追求逐 stage 的耗时统计
- 不要求 `Agent` 识别返回的是哪一条 lane
- 更强调数据继续向出口推进
- 更适合连续处理和更平滑的结果消费

在这个语义下，`Agent` 只需要知道提交批次数量和最终结果。

## 4. Lane 语义

每一次提交都可以形成一条 lane。

`Agent` 只知道：

- 自己提交了多少轮
- 自己当前持有的算法对象是什么

`Agent` 不需要知道：

- lane 具体编号
- lane 最终以哪一条身份返回
- lane 在内部如何被推进

如果多个 `Agent` 挂载了同一个流水线，调度中心需要区分 owner。
也就是说，lane 和流水线状态必须绑定提交者，不能只靠流水线名字。

## 5. 流水线推进

流水线推进由调度中心负责。

### 5.1 强制同步推进

强制同步下，调度中心会记录：

- 单 stage 耗时
- 整体耗时
- 当前阶段的运行理由

这些数据会被保存在 runtime state 里，再由 `debugTool` 读取。

### 5.2 非强制同步推进

非强制同步下，调度中心只保留执行所需的核心状态。

它不再强调耗时统计，而是让 lane 按流水线继续前进，直到到达出口。

## 6. 统计能力

统计能力只给 `debugTool`。

### 6.1 强制同步总耗时

需要回答的问题是：

> 这套流水线从输入到执行完毕，总共花了多少时间？

输出内容建议包括：

- 总耗时
- 开始时间
- 结束时间

### 6.2 强制同步逐 stage 耗时

需要回答的问题是：

> 强制同步下，每一个 stage 花了多少时间？

输出内容建议包括：

- stage 名称
- stage 顺序
- 单次耗时
- 累计耗时
- 最近一次运行理由

## 7. 调度中心职责

`AlgorithmScheduler` 需要负责：

- 持有流水线定义
- 持有流水线 runtime
- 区分 lane owner
- 维护提交和 tick 关系
- 维护强制同步统计
- 维护非强制同步推进
- 把结果和状态回传给上层

`Agent` 不需要知道调度中心内部有哪些子结构。

## 8. 推荐的数据结构

建议保留这些概念：

- `PipelineRuntimeState`
- `PipelineLaneRuntimeState`
- `PipelineStageRuntimeStat`
- `owner_agent_name`

其中：

- `PipelineRuntimeState` 持有整条流水线的状态
- `PipelineLaneRuntimeState` 持有一条 lane 的运行状态
- `PipelineStageRuntimeStat` 只在强制同步里保留
- `owner_agent_name` 用来区分多个 `Agent` 对同一流水线的挂载

## 9. 小结

流水线算法现在应该被理解为：

- `Agent` 只提交算法对象和资源批次
- `AlgorithmScheduler` 负责把资源组织成 lane 并推进流水线
- 强制同步提供可视化和统计
- 非强制同步提供吞吐和连续推进
- `debugTool` 只消费强制同步暴露出来的统计数据
