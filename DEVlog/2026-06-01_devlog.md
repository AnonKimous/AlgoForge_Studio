# 2026-06-01 Dev Log / 第二次修改记录

This entry records the second round of June 1 changes in the current working tree.

## 1. What changed

- Reframed the runtime around `entity` instead of “instance” terminology.
- Kept the shared random-motion preset as a single entity that mounts both render and physics roles.
- Made the entity creation UI default-filled so it is ready to create and run immediately.
- Changed rendering to use the algorithm-side vertex array view instead of treating the source `Mesh` as the render primitive.
- Raised the random drift radius UI ceiling to `100`, so the effect remains visible during manual testing.

## 2. Current runtime model

- `entity_interaction` owns the manual entity composer UI.
- A shared entity can bind render and physics roles at once.
- Physics updates the active vertex array; rendering reads the current vertex array and topology arrays.
- The UI defaults to a shared entity preset and starts the work on creation.

## 3. Validation

- `min_vulkan_win32` builds successfully after the latest changes.
- The remaining build output is the existing Eigen `C4819` warning and the unrelated `pwsh.exe` post-build message.

## 4. Terminology and instance-level composition

- In the project docs, Chinese `实例` should be read as `entity` unless a file explicitly says otherwise.
- The instance-level algorithm model is: small algorithms stay in `algorithm_library`, while `entity` composes them through ordered bindings, aliases, and container routes.
- The final entity-level compliance descriptor is what gets passed to `algorithm` / `algorithm_mng` for container allocation.
- A helper composer script is still the preferred way to build and rename merged descriptors before wiring them into an entity.
