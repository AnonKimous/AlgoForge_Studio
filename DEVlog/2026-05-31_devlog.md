English: The project now clearly follows the dependency chain `common_data -> runtime_systems -> messaging -> algorithm -> algorithm_support -> service_domains -> interaction_analysis -> agents -> app_orchestration`. The important progress today is that this is no longer just a documentation guideline; it is reflected directly in `CMakeLists.txt`, static library boundaries, and source ownership.

English: The most visible change today was large-scale cleanup of legacy transition code. Old layers such as `foundation`, `data_protocol`, `decomposition`, and `service_domains/physics`, along with several controller / relay / manager transitional components, were removed. This shows the project is intentionally ending its old-architecture compatibility phase. The current diff touches **61 files**, with about **546 insertions and 5313 deletions**, which is clearly a **subtractive refactor**.

## 4. 共享数据层统一为 `common_data` / Shared Data Was Unified Under `common_data`

中文：原先分散在 `foundation` 和 `data_protocol` 的基础类型与共享协议，已经集中到 `common_data`。这里现在承载 `Mesh`、`viewport_transform`、`input_state`、`window_handle`、交互状态、物理状态和向量类型等跨层公共结构，减少了命名漂移和重复抽象。

English: Basic types and shared protocol data previously scattered across `foundation` and `data_protocol` were consolidated into `common_data`. This layer now holds shared structures such as `Mesh`, `viewport_transform`, `input_state`, `window_handle`, interaction state, physics state, and vector types, reducing naming drift and duplicate abstractions.

## 5. 算法层被收束到“实现 + 合规模型” / Algorithm Layer Was Reduced to “Implementation + Contract”

中文：`algorithm` 层今天更明确地只保留两类职责：一类是算法执行实现，另一类是算法合规描述（compliance contract）。当前保留的主线算法是 CPU 侧的 `corotated_cpu` 与 GPU 侧的 `physics_convolution_demo`，并通过 `PhysicsAlgorithmPipeline_Run` 按 `PhysSolverKind` 做统一分发。`algorithm_manager`、`cpu_job_scheduler`、三角朝向旧算法和 legacy contract 都被清掉了。

English: The `algorithm` layer is now much more focused, keeping only two responsibilities: algorithm execution and algorithm compliance descriptors. The active paths are the CPU-side `corotated_cpu` and the GPU-side `physics_convolution_demo`, both dispatched through `PhysicsAlgorithmPipeline_Run` via `PhysSolverKind`. Legacy pieces such as `algorithm_manager`, `cpu_job_scheduler`, triangle-orientation code, and old contracts were removed.

## 6. `algorithm_support` 成为算法干预的唯一打包入口 / `algorithm_support` Became the Single Packaging Entry for Algorithm Intervention

中文：今天一个很关键的收敛点，是把“算法干预”整理成 `algorithm_support` 层负责的标准 packet 编解码流程。`AlgorithmSupportManager` 现在负责把 intervention descriptor 打成 `IoBufferPacket`，再从 packet 解回结构化描述。这样做的意义是，agent 或上层逻辑不再直接揉业务二进制格式，而是统一走算法支持层。

English: A key architectural consolidation today was making `algorithm_support` the standard encode/decode path for algorithm interventions. `AlgorithmSupportManager` now builds intervention descriptors into `IoBufferPacket`s and decodes them back into structured data. This means agents and upper layers no longer manipulate business-level binary payloads directly; they go through a single algorithm support path.

## 7. `messaging` 被压缩成纯传输层 / `messaging` Was Shrunk Into a Pure Transport Layer

中文：`messaging` 的角色被明显收窄，只保留 buffer / fast channel 的分配、发布、读取和 packet 拷贝能力。它不再承担协议解释、业务路由或高层状态同步。这是今天最成熟的一次边界清理之一，因为它让消息层回到“搬运数据”而不是“理解业务”。

English: The `messaging` layer was intentionally narrowed to transport responsibilities only: allocating buffers and fast channels, publishing, reading, and copying packets. It no longer handles protocol interpretation, business routing, or high-level state synchronization. This is one of today’s strongest boundary cleanups, because messaging is now back to moving data instead of understanding business logic.

## 8. 物理逻辑由 `PhysAgent` 直接组织 / Physics Flow Is Now Organized Directly by `PhysAgent`

中文：旧的 `PhysManager` 被移除后，`interaction_analysis::PhysAgent` 直接承担物理求解生命周期：初始化 rest state、构建 `PhysicsAlgorithmRequest`、维护速度缓存、按固定步长 `1/120s` 推进、支持 `Run / Pause / StepOnce / Reset`，并把 solver 输出写回 mesh。这个变化让物理运行链路更短，也更容易追踪。

English: With the old `PhysManager` removed, `interaction_analysis::PhysAgent` now directly owns the physics lifecycle: initializing rest-state data, building `PhysicsAlgorithmRequest`, maintaining velocity caches, advancing at a fixed `1/120s` step, supporting `Run / Pause / StepOnce / Reset`, and writing solver output back to the mesh. This makes the physics execution path shorter and easier to reason about.

## 9. 主循环被简化为更直接的协作关系 / The Main Loop Was Simplified Into a Clearer Collaboration Pattern

中文：`main.cpp` 现在的主线非常清楚：SDL 初始化 -> mesh 载入/生成 -> `WindowAgent` / `SceneViewAgent` / `RenderAgent` / `PhysAgent` 初始化 -> 每帧同步 render run state 到 physics -> physics tick -> glue 组装 UI 状态 -> renderer 输出 frame result -> glue 把 reset/step 结果反馈给 physics 和 mesh。整体协作关系比之前更扁平。

English: The flow in `main.cpp` is now much clearer: SDL init -> mesh load/generation -> initialize `WindowAgent`, `SceneViewAgent`, `RenderAgent`, and `PhysAgent` -> sync render run state into physics every frame -> physics tick -> build UI state through glue -> renderer returns a frame result -> glue applies reset/step requests back to physics and mesh. The collaboration pattern is noticeably flatter than before.
