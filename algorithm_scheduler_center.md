# AlgorithmScheduler 调度中心说明

## 1. 作用

`AlgorithmScheduler` 是项目里统一的算法调度入口。
它不直接替代底层执行器，而是把上层看到的提交、挂载、路由、卸载和统计入口集中到一层。

它负责把这些事情统一起来：

- `Agent` 只面对 `AlgorithmScheduler`
- 普通算法、流水线算法、CPU 提交、GPU 提交都走同一套入口
- 调度中心再把请求转给 `runtime_systems`
- `runtime_systems` 继续负责真正的执行

## 2. 调度中心要解决什么

调度中心主要解决三类问题：

### 2.1 统一入口

同一套提交接口覆盖：

- 普通算法
- 流水线算法
- CPU 路径
- GPU 路径
- debugTool 统计读回

### 2.2 生命周期

同名流水线可以被多个 `Agent` 使用。
调度中心负责：

- 挂载
- 引用计数
- 共享状态
- 卸载回收

### 2.3 路由

流水线算法的运行态不能散落在 `Agent` 里。
调度中心要保存对应的运行时状态，再把它路由给底层执行路径。

## 3. 责任边界

### 3.1 `Agent`

`Agent` 只保留算法视角：

- 提交算法对象
- 发起 tick
- 收取结果
- 不关心调度细节

### 3.2 `AlgorithmScheduler`

`AlgorithmScheduler` 负责：

- 统一接入口
- 维护流水线 runtime
- 维护 lane 归属
- 维护生命周期
- 维护 debugTool 可见的统计数据
- 把请求转给底层执行器

### 3.3 `runtime_systems`

`runtime_systems` 负责：

- CPU / GPU 真正执行
- 线程调度
- 资源流转
- 运行时数据更新

## 4. 与流水线的关系

流水线算法由调度中心托管。
对于流水线算法，调度中心持有：

- 流水线定义
- lane 列表
- stage 状态
- owner 归属
- 运行统计

调度中心不把“谁提交了什么”交给 `Agent` 自己记忆。
`Agent` 只保留自己提交了多少轮资源的视角。

## 5. 两种同步语义

当前对外只有两种语义：

### 5.1 强制同步

强制同步用于：

- 需要逐 stage 观察执行过程
- 需要总耗时统计
- 需要 debugTool 展示运行轨迹

它关心的是：

- 单 stage 花了多久
- 整条流水线花了多久

这些数据只在 debugTool 侧暴露。

### 5.2 非强制同步

非强制同步用于：

- 更偏吞吐的流水线执行
- 不要求用户追踪具体 lane 身份
- 不需要暴露耗时统计

在这个语义下：

- `Agent` 只知道自己提交了多少轮资源
- `Agent` 不关心返回的是哪一条 lane
- 调度中心只负责让数据继续往出口推进

## 6. 调度中心输出

调度中心对外需要维持这些能力：

- 算法挂载
- 算法提交
- 流水线提交
- tick
- 卸载
- debugTool 统计读回

## 7. 一句话总结

`AlgorithmScheduler` 的目标不是再造一套执行器，而是把“提交、挂载、路由、生命周期、统计”收拢到同一层，让 `Agent` 只保留算法视角，让底层只保留执行视角。
