CPU Pipeline Executor 对齐设计文档

目标
1. 参考 GPU 侧 `AlgorithmGpuExecutor` 的形状，在 `runtime_systems` 内新增 `AlgorithmCpuExecutor`。
2. 以后所有 CPU 算法执行都先进入 `AlgorithmCpuExecutor`，再由它决定如何把工作挂到 jobs 系统。
3. CPU pipeline 遵守和当前 GPU pipeline 一致的两段提交流程：
   - 第一段：挂载 pipeline
   - 第二段：提交资源/描述符
4. CPU pipeline 的最终 owner 从 `agent` 迁移到 `runtime_systems::AlgorithmCpuExecutor`。

为什么不直接让 job_system 持有 pipeline
1. `job_system` 只适合做任务执行与线程分发，不适合持有 pipeline 的业务生命。
2. pipeline 需要持有的状态很多：
   - pipeline 名称
   - stage 数量与 stage 顺序
   - circular / non-circular 模式
   - lane 状态
   - standard container
   - mandatory stageBuffer 槽位
   - pending stage0 提交
   - stall 统计与调试信息
3. 上述状态如果直接塞进 `job_system`，会让调度层和执行层耦合，后续调试和热重载都会更脏。
4. 所以正确分层应为：
   - `agent` 负责上层挂载/提交/tick 入口
   - `runtime_systems::AlgorithmCpuExecutor` 负责 CPU 执行态与 pipeline owner
   - `job_system` 负责 CPU job 执行

对齐 GPU 侧后的目标分层
1. GPU 侧
   - `AlgorithmGpuExecutor` 持有 GPU 执行缓存
   - 上层提交算法对象与容器给 executor
2. CPU 侧
   - `AlgorithmCpuExecutor` 持有 CPU pipeline runtime state
   - 上层提交算法对象或 pipeline 给 executor
   - executor 内部再把单个 stage 的执行挂到 jobs 系统

CPU executor 最终应承担的职责
1. 普通 CPU 算法执行入口
   - 接收算法对象
   - 挂 jobs
   - 回填 `AlgorithmToAgentSignal`
   - 回填 debug state
2. CPU pipeline owner
   - 挂载 pipeline
   - 保存 pipeline 元数据与 runtime state
   - 接收 stage0 资源提交
   - 处理 circular / non-circular tick
   - 统计每个 stage 耗时与原因
3. stage 级执行单元
   - ingress bridge
   - 算法执行
   - egress bridge
   - stage debug capture

CPU pipeline 最终 tick 语义
1. executor 持有整条 CPU pipeline 的 runtime state。
2. 一次 tick 时，由 executor 判定哪些 stage 本轮可执行。
3. 每个可执行 stage 作为一个独立 job 单元执行：
   - bridge ingress
   - stage execute
   - bridge egress
4. 同一条 lane 的 stage 仍然遵守前后依赖，不能乱序并行。
5. circular pipeline 的 stage-1 回灌 stage0，由 executor 内部处理，不交给 agent 临时拼。

这次改造的落地边界
第一阶段先完成：
1. 新增 `runtime_systems::AlgorithmCpuExecutor`。
2. 所有 CPU 普通算法执行入口改为先进 `AlgorithmCpuExecutor`。
3. CPU pipeline 在 mount/unmount 时向 `AlgorithmCpuExecutor` 注册/注销元数据。
4. `AlgorithmCpuExecutor` 先持有最小 pipeline 注册信息：
   - pipeline 名称
   - 根 stage 名
   - stage 数量
   - submission mode
   - mandatory stageBuffer 槽位

第二阶段继续完成：
1. 把 `agent` 里的 `PipelineRuntimeState` 迁入 `AlgorithmCpuExecutor`。
2. 把 pending stage0 submission 迁入 `AlgorithmCpuExecutor`。
3. 把 CPU pipeline 的 tick 主循环从 `agent` 挪到 `AlgorithmCpuExecutor`。
4. 把 bridge debug、stall report、lane 管理都收口到 `AlgorithmCpuExecutor`。

当前代码要点
1. 现状中，GPU executor 已在 `runtime_systems` 中独立存在。
2. 现状中，CPU 普通算法仍直接走 `job_cpu::Execute(...)`。
3. 现状中，CPU pipeline runtime state 仍保存在 `agent::Agent` 内部。

执行原则
1. 不改算法容器底层 `vn/an` 规则。
2. 不改 pipeline 的两段提交语义。
3. 不写静默失败，状态不一致直接报错/断言。
4. 先做 CPU executor 收口，再继续迁 pipeline owner。
